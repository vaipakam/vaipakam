# Test Cases - 2026-04-25 Review Findings

- [x] **00001: High: Permit2 paths do not validate the signed token against the expected protocol asset.**

- VPFIDiscountFacet.sol (line 384) calls \_prepareDeposit(amount) to resolve the configured VPFI token, but then calls LibPermit2.pull(...) without checking permit.permitted.token == vpfi at line 390 (line 390). Same pattern exists in offer creation at OfferFacet.sol (line 184) and offer acceptance at line 781 (line 781). A user could submit a valid Permit2 signature for a different ERC-20; Permit2 would faithfully pull that token, while Vaipakam records the offer/deposit as if the expected asset was funded. This directly contradicts the docs’ “pre-validates asset shape / validate VPFI asset” requirement and can create unfunded offers or fake VPFI escrow deposits.

- [x] **00002: High: Pyth is not removed end-to-end in the frontend.**

- useWriteWithPythUpdate.ts (line 96) still calls getPythEndpoint, and pyth.ts (line 1) still contains the Hermes/update transaction flow. The frontend ABI also still exposes removed Pyth functions in OracleAdminFacet.json (line 28). Since the docs now say Pyth was removed and the current path is Chainlink + Tellor/API3/DIA, these stale surfaces are a regression risk and any future use of the hook will fail against the current Diamond.

- [x] **00003: Medium: Blockaid API key is still browser-exposed instead of server-side.**

- The docs require the Blockaid key to live server-side behind a worker proxy, but useTxSimulation.ts (line 55) reads VITE*BLOCKAID_API_KEY, and lines 129-134 (line 129) call Blockaid directly with X-API-KEY from the browser. Any VITE*\* value is shipped to clients, so this leaks the scanner key and bypasses the documented worker-internal proxy model.

That means a user buying on one origin chain consumes allowance for all chains, contrary to README.md and TokenomicsTechSpec.md.

- [x] **00006: High: Mirror-chain Buy VPFI allowance reads the wrong wallet bucket.**

- BuyVPFI.tsx (line 298) correctly reads fixed-rate config from the canonical chain for mirror buys, but useVPFIDiscount.ts (line 121) still calls getVPFISoldTo(user), which returns the canonical chain's local bucket. The docs require allowance to be chain-local per origin chain, so mirror buys should read getVPFISoldToByEid(user, activeChain.lzEid) while still reading caps/rate from canonical. Otherwise the UI can show remaining allowance when the user's origin-chain cap is already spent, leading to Base-side refund/revert.

- [x] **00007: Medium: Blockaid fail-soft handling is incomplete for rate limits and upstream errors.**

- The docs say Blockaid unavailability should collapse to preview-unavailable, but scanProxy.ts (line 51) returns 429 rate-limited, and useTxSimulation.ts (line 139) only downgrades 503 and network errors to unavailable. All other non-OK statuses become error, which makes rate limits or upstream outages render as a warning in SimulationPreview.tsx (line 58), contrary to the fail-soft requirement.

- [ ] **00017: High: Cross-chain interaction reward source configuration omits Base from the global denominator.**

- TokenomicsTechSpec.md defines interaction rewards using a protocol-wide daily denominator across all chains, with Base acting as the canonical aggregator. ConfigureRewardReporter.s.sol describes REWARD_EXPECTED_SOURCE_EIDS as mirror EIDs only, and BaseSepoliaDeploy.md sets it to 40161,40231,40232,40267. RewardAggregatorFacet.\_finalizeAndWrite only sums the configured expectedSourceEids, so Base local closeDay totals are not included unless Base's own localEid is also configured. This excludes Base interest from dailyGlobalInterestRepaid and can distort every chain's reward allocation.

- [ ] **00018: High: Mirror interaction reward budgets are not bridged during finalization.**

- TokenomicsTechSpec.md says the interaction reward pool is held on Base, Base computes each mirror chain's budget, and the Base treasury bridges that budget to each mirror through OFT before mirror users claim locally. RewardAggregatorFacet.\_finalizeAndWrite only totals reports, stores global values, and broadcasts the result; it does not bridge or escrow per-chain VPFI budgets. InteractionRewardsFacet.claimInteractionRewards then transfers VPFI from the local Diamond balance, so mirror claims can fail from an unfunded local balance even when the Base pool has enough budget.
