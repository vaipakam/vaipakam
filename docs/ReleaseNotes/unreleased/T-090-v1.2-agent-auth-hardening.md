## Thread — Agent endpoint on-chain commit preflight (T-090 v1.2 #428, PR #<n>)

Closes #428. Hardens the agent worker's `POST /intent/fusion/post` endpoint against shape-valid-but-fake commits that could spend the shared 1inch Fusion API quota on never-existed orders.

The endpoint's pre-existing gates — `Origin` allow-list, per-IP rate-limit binding, maker/receiver-bound-to-diamond, exact-length hex payload validation, half-activated fail-closed — bounded the noise from misconfigured browser callers and limited the spend rate. None of them proved that the commit actually happened on-chain. A non-browser caller spoofing an allowed Origin could still push shape-valid commits whose `orderHash` was never registered, burning the Fusion API quota at the upstream rejection.

This PR adds an on-chain commit preflight at the agent boundary. After the per-chain RPC binding lookup, the handler fetches the `commitTxHash` receipt and verifies it contains a `SwapToRepayIntentCommitted` event log emitted by the canonical Vaipakam diamond on the request's `chainId`, with the event's indexed `orderHash` field matching the body's `orderHash`. Any mismatch returns a discriminated 400 (`commit-tx-not-successful`, `orderhash-not-in-commit-tx`) before the Fusion `fetch` runs.

Cost: one `eth_getTransactionReceipt` per request. Bounded above by the existing rate-limit binding (30 req/60s). The RPC URL is read from the same per-chain secrets the buy-watchdog uses (`RPC_BASE`, `RPC_ETH`, etc.). When the RPC URL isn't bound (operator-pre-activation, test environments, chains the worker doesn't hold a secret for), the preflight skips with a warn log and the request falls through to the existing gates + Fusion's server-side ERC-1271 staticcall, which is the final on-chain backstop against unregistered orderHashes.

No dapp-side changes — the request shape is unchanged and the panel's response handling already surfaces 4xx errors from the agent to the borrower.
