## Thread — T-090 Swap-to-Repay (PRs #390 / #391 / #402)

A borrower can now repay a Vaipakam ERC-20 loan by swapping their pledged collateral asset into the loan's principal asset and applying the proceeds to the settlement waterfall, all in a single transaction. Before this thread, the canonical way to repay a loan where the borrower happened to hold the wrong asset was a four-step dance: withdraw collateral to wallet, externally swap on a DEX, re-deposit the principal asset, then call the regular repay entry point. Each step leaked gas, slippage, and custody surface area — and the borrower had to manage the slippage themselves on every leg.

The new surface fuses the swap and the settlement into one atomic call. There are two modes — a full close that drives the loan to Repaid and respects the loan's full-term-vs-pro-rata interest choice, and a partial reduction that requires the offer to have been created with partial-repay opt-in. Both modes reuse the platform's existing 4-DEX swap-failover primitive (the same 0x v2 / 1inch v6 / Uniswap V3 / Balancer V2 adapter set the HF-liquidation path uses), so there is no new adapter wiring or governance surface. A new protocol-config knob caps the borrower's realized slippage at 3% by default — tighter than the 6% liquidation cap because the borrower is not on an adversarial clock and can wait for better price action.

Settlement waterfall mirrors the regular repay path exactly: lender (in full close) gets their settlement entitlement deposited to their vault, treasury receives the standard cut on interest and late-fee, the borrower's residual pledged collateral records as a claim, and any favorable-quote surplus principal is transferred directly to the current borrower-position NFT holder's wallet. The full path also performs all the proper-close housekeeping the regular path does: the borrower LIF VPFI rebate settles, any live prepay-listing on the loan clears cleanly, both position NFTs flip to LoanRepaid status, and the Phase-2 reward accrual closes with the borrower's clean-close credit.

Authority is the **current borrower-side position NFT owner**, not the borrower of record on the loan struct — claim rights travel with the NFT and the swap-to-repay surface respects that. A degenerate case where the lender also holds the borrower NFT is blocked by the same lender-self-repay guard the regular repay path uses. Total swap failure (every adapter in the keeper-supplied try-list reverts) reverts the whole transaction with no soft fallback; the borrower can retry with better routing.

The Activity and Loan Details Timeline surfaces both render the new event kinds — full close as a `success` row with the same checkmark as the canonical repay paths, partial reduction as `info` with collateral-swapped / principal-received / principal-retired amounts formatted as token amounts. The `useIndexedLoans` near-realtime catch-up treats a swap-to-repay full close as a terminal close just like `LoanRepaid` / `LoanDefaulted`, so the active-loans surface filters out a swap-to-repay'd loan during the brief window between the on-chain confirmation and the indexer worker's next scan. The fallback browser RPC log scanner handles both kinds too, so worker-down users see the rows the same way.
