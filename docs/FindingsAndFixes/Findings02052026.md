## New findings

- [ ] **00037: Low: Late-fee schedule has no precise boundary regression test (1% day-1 + 0.5%/day, capped at 5%).**

- `LibVaipakam.calculateLateFee` (`libraries/LibVaipakam.sol:2284-2300`) implements the schedule documented in ProjectDetailsREADME §6 "Late Fees" (line 519): `1% on the first day past due, +0.5% each subsequent day, capped at 5%`. Existing coverage exercises the late-fee BRANCH but not the schedule's numerical boundaries — `RepayFacetTest.testCalculateRepaymentAmountERC20LateFee` (line 1023) only asserts `due > 1000` (some late fee was applied), and `Scenario1_ERC20LendingLifecycle` warps past grace but doesn't read the late-fee figure separately. A regression that flipped the cap to 50% or the per-day increment to 0.05% would slip past every test in the suite. Concrete gap: no existing test asserts `calculateLateFee` returns the exact basis-point value at the documented boundary days (day 1 = 100 bps, day 2 = 150 bps, day 8 = 500 bps, day 9 = 500 bps cap-clamped, day 100 = 500 bps still capped). Recommendation: add a `LateFeeScheduleTest.t.sol` that walks `calculateRepaymentAmount(loanId)` minus the known principal+interest, asserting the implied lateFee matches the documented bps schedule across the boundary table. Single shared loan, time-warp per case, ~6 assertions.

- [ ] **00036: Low: Cross-mechanism invariants in TokenomicsTechSpec §7b lack a unified regression test surface.**

- TokenomicsTechSpec.md §7b (lines 473-478) lists four cross-mechanism invariants the protocol must uphold:
  1. every escrow-balance change must refresh the user's staking position AND lender-side time-weighted discount tracker
  2. governance APR / threshold / discount-percent changes apply prospectively only
  3. dormant holders don't lose accrual during governance windows
  4. no late single-moment action inflates rewards/discounts for prior periods spent at lower balance/tier

Each is exercised individually inside `LibStakingRewardsTest`, `LibVPFIDiscountTest`, `StakingAprPipelineTest`, etc., but the combined SHAPE of "every escrow mutation site simultaneously refreshes both trackers" isn't asserted as a single invariant. New code paths that introduce an escrow mutation (e.g. a future facet) could silently miss either rollup; the invariant test would catch that at PR time. Recommendation: add an invariant-style test under `test/invariants/CrossMechanismInvariants.invariant.t.sol` that fuzzes a sequence of escrow mutation primitives (deposit / withdraw / claim-to-escrow / fee-driven deduction) and asserts after each step that both the staking accumulator AND the user's discount accumulator timestamps moved to `block.timestamp`. Out of scope for a quick fix; queue as a Phase 1 hardening task.

- [ ] **00035: Low: Treasury Recycling Rule split (38/38/24) has no on-chain assertion test.**

- TokenomicsTechSpec.md §9 (lines 546-554) AND ProjectDetailsREADME §10 prescribe an automated `38% ETH / 38% wBTC / 24% retain` split for VPFI received as fees. Even if the recycling loop itself is operator-driven (per finding 00028), the RATIO numbers should be exposed as named library constants and asserted in a test so spec changes (e.g. tune to 35/35/30) require updating the code in lockstep with the doc. Today the literal numbers `38, 38, 24` don't appear in `LibVaipakam.sol` — they exist only in the spec text. A coverage test that reads the constants and asserts `bps_eth + bps_wbtc + bps_retain == BASIS_POINTS` would force any future split change to land in code first. Pre-requisite to landing the recycling logic itself; a 5-line constant + 1 assertion is the smallest defensible step.

- [ ] **00034: Low: Per-wallet-per-eid fixed-rate VPFI cap segregation has no direct invariant test.**

- TokenomicsTechSpec.md §8 (lines 489, 508) is explicit that the per-wallet fixed-rate cap is keyed on `(buyer, originEid)` — buying up to the cap on chain A must NOT consume the wallet's allowance on chain B. The implementation uses `Storage.vpfiFixedRateSoldToByEid[buyer][originEid]` (LibVaipakam.sol:933) and the per-eid path is exercised in `VPFIDiscountFacetTest` and `VPFIBuyAdapterPaymentTokenTest`, but I couldn't find a test that exhausts the cap on one `originEid`, then asserts a second buy with a DIFFERENT `originEid` (same wallet) succeeds without pulling from the first eid's bucket. The spec is explicit enough that a regression introducing a single global counter (instead of per-eid) would be a real bug. Recommendation: extend `VPFIDiscountFacetTest` with a "buys cross-eid don't share the cap" two-call test against the canonical receiver. Small surface; ~30 lines.

- [ ] **00033: Low: ProjectDetailsREADME §2 (Loan Terms) describes "Configurable from 1 day to 1 year" but the cap floor is now 7 days, not 1 day.**

- 00025 (resolved 2026-05-01) introduced `MIN_OFFER_DURATION_DAYS_FLOOR = 7` as the lower bound for the admin-tunable `cfgMaxOfferDurationDays()` setter. That's the lower bound on the CAP setter (operator can't lock the protocol to "1 day max"); separately, the offer creation path itself only enforces `params.durationDays != 0`, so a 1-day or 6-day offer still creates fine. So the spec wording remains technically accurate (1 day IS configurable from a borrower / lender perspective), but a reader diffing the spec against `LibVaipakam.MIN_OFFER_DURATION_DAYS_FLOOR` could be confused into thinking the floor applies to the offer parameter, not the admin cap. Recommendation: a one-line spec clarification — "the admin-tunable max-duration ceiling can be set anywhere in `[7, 1825]` days; offer creation accepts any non-zero duration up to the configured ceiling." Doc-only fix.

## Pattern observations across the audit

A few recurring patterns surfaced both in the new findings and the
re-validation pass — worth capturing here for the next session:

1. **Spec ↔ event-shape drift** (00027 + 00026). The spec frequently
   asserts an event/function shape with specific named parameters; the
   implementation has the parameter set but renames or omits the
   timestamp / dayId. A small "spec-event-parity" test that consults a
   structured registry of `(facet, event, signature)` triples and
   asserts the deployed Diamond exposes them would catch all current
   and future drift in one shape. Out of scope for a quick fix.

2. **Parallel selector lists for the same facet** (00011 + 00012). Two
   different deploy scripts each maintain their own hand-listed selector
   set per facet, which drift independently as new methods land. The
   pattern that closed 00010 — a single
   `LibAccessControl.grantableRoles()` view consulted by both the
   library init AND the deploy script — is the right shape here too.
   Move every facet's selector list into a `LibFacetSelectors.X()`
   helper consulted by `DeployDiamond`, `ReplaceStaleFacets`,
   `RedeployFacets`, and `HelperTest`. Single source of truth.

3. **Documented invariant ↔ no targeted regression test** (00037 +
   00036 + 00035 + 00034). The protocol's most-emphasized formulas and
   invariants (late-fee schedule, cross-mechanism rollup invariants,
   treasury recycling ratios, per-eid cap segregation) have indirect
   coverage but not a focused boundary table that asserts the documented
   numbers. These are the easiest tests to write — concrete spec
   numbers, narrow setup, fast to run — and the most valuable for
   long-term spec-as-source-of-truth maintenance.

4. **Spec internal contradictions** (00030 between §5a and §6b). When
   two sections of the same spec doc contradict each other, the choice
   of which to honour falls implicitly on whoever last touched the
   relevant code — usually invisibly. A short-form "spec-conflict log"
   in `docs/FunctionalSpecs/` listing every known conflict + the chosen
   reading + a target resolution date would force these into the open
   so they get reconciled instead of accreting. Out of scope for a
   quick fix; queue as a process improvement.
