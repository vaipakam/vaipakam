- [ ] **00008: High: Fixed-rate VPFI buy still relies on a pre-funded Diamond sale reserve.**

- TokenomicsTechSpec.md says the fixed-rate purchase program must not rely on a pre-minted sale reserve. VPFIDiscountFacet.sol still documents that the Diamond sells VPFI from its own balance and that ops must fund it during canonical deploy, then both direct and bridged buys transfer from the Diamond balance and revert with VPFIReserveInsufficient when on-hand VPFI is below the requested amount. This can halt fixed-rate buys even when caps remain available, and it contradicts the documented mint/release-on-receipt model.

- [ ] **00007: High: Cross-chain interaction reward source configuration omits Base from the global denominator.**

- TokenomicsTechSpec.md defines interaction rewards using a protocol-wide daily denominator across all chains, with Base acting as the canonical aggregator. ConfigureRewardReporter.s.sol describes REWARD_EXPECTED_SOURCE_EIDS as mirror EIDs only, and BaseSepoliaDeploy.md sets it to 40161,40231,40232,40267. RewardAggregatorFacet.\_finalizeAndWrite only sums the configured expectedSourceEids, so Base local closeDay totals are not included unless Base's own localEid is also configured. This excludes Base interest from dailyGlobalInterestRepaid and can distort every chain's reward allocation.

- [ ] **00006: High: Mirror interaction reward budgets are not bridged during finalization.**

- TokenomicsTechSpec.md says the interaction reward pool is held on Base, Base computes each mirror chain's budget, and the Base treasury bridges that budget to each mirror through OFT before mirror users claim locally. RewardAggregatorFacet.\_finalizeAndWrite only totals reports, stores global values, and broadcasts the result; it does not bridge or escrow per-chain VPFI budgets. InteractionRewardsFacet.claimInteractionRewards then transfers VPFI from the local Diamond balance, so mirror claims can fail from an unfunded local balance even when the Base pool has enough budget.

- [ ] **00005: Medium: The service worker can cache same-origin quote responses.**

- WebsiteReadme.md requires production service workers to cache only the static shell and explicitly bypass dynamic RPC, subgraph, /quote/\*, and transaction-preview responses. frontend/public/sw.js bypasses only Vite internals and /api/ for same-origin GET requests, then applies stale-while-revalidate caching to successful same-origin responses. If quote endpoints are deployed under the same origin, stale quote data can be served after market or liquidity conditions change.

- [ ] **00004: Medium: The PWA Buy VPFI shortcut points to a route that does not exist.**

- WebsiteReadme.md expects the PWA manifest to expose a Buy VPFI shortcut. frontend/public/manifest.json points that shortcut to /app/buy, but App.tsx registers the actual buy route as /app/buy-vpfi. Installed users who launch the shortcut will land on an unmatched route instead of the public VPFI purchase screen.

- [ ] **00003: Medium: The liquidation UI promises fallback submission but disables the submit path.**

- WebsiteReadme.md says liquidation routing should submit the available ranked route list and route failures into the documented fallback flow. LiquidateButton.tsx displays "The loan will route to FallbackPending on submit" when quotes are unavailable, but handleClick exits unless quotes.calls.length is greater than zero and the Liquidate button is disabled unless status is ready. A no-quote loan therefore cannot be submitted from the frontend despite the UI text saying it will trigger fallback.
